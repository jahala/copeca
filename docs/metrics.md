# Metrics: cost per correct answer

Copeca's headline metric is **cost per correct answer**: the expected dollar
cost before getting one right answer. Every saving or regression is judged
through this lens.

---

## Cost-per-correct definition

```
cost_per_correct = total_cost_usd / correct_count
```

`total_cost_usd` is the vendor's billed cost when the runner reports it (the real
bill — it reflects cache hits, cache TTL, service tier, and discounts), summed
across all runs in a mode. copeca also records `computed_cost_usd` (Σ tokens × the
pricing table) as a reproducible, provider-neutral cross-check; when a runner
reports no cost, the computed figure is the fallback.
`correct_count` is the number of runs where the agent got the answer right.

A mode with 100 runs at $0.10 each and 50 correct answers has a
cost-per-correct of $0.20.

---

## Delta: the comparison that matters

Copeca compares modes, not absolutes. The delta is:

```
delta = (experimental_cost_per_correct - baseline_cost_per_correct)
      / baseline_cost_per_correct × 100%
```

A negative delta means the experimental mode is cheaper per correct answer.
A positive delta means it is more expensive.

The delta is the first number in every copeca report. Individual mode
cost-per-correct values are shown, but the delta is the headline.

---

## Bootstrap 95% confidence interval

The delta CI is a paired bootstrap over per-task deltas:

1. For each task, compute cost-per-correct for both modes and their percentage
   delta: `((cpc_exp - cpc_base) / cpc_base) × 100`.
2. Resample the resulting list of per-task deltas with replacement and compute
   the mean of each resample. Repeat 10,000 times.
3. The 95% CI is the 2.5th and 97.5th percentiles of those resampled means.

Pairing the two modes within each task removes cross-mode variance and makes
the CI sensitive to the within-task difference rather than to task difficulty.
Overlapping per-mode CIs do not mean the delta is non-significant — the delta
CI is computed directly from the paired per-task deltas.

---

## Why cost-per-correct is not accuracy

Accuracy alone is a trap: a tool that slashes tokens by 90% but gets the
answer wrong 20% more often can still have a worse cost-per-correct than the
baseline. The metric punishes both waste and wrongness:

| Scenario | Total cost | Correct | Cost-per-correct |
|---|---|---|---|
| Baseline | $1.00 | 8/10 | $0.125 |
| Tool (saves 90% tokens) | $0.10 | 6/10 | $0.017 |
| Tool (saves 90% tokens, 50% wrong) | $0.10 | 4/10 | $0.025 |

The third row's cost-per-correct looks great at $0.025 — but the tool is
wrong 60% of the time. This is copeca's central insight: savings that hurt
accuracy can still improve cost-per-correct. Whether that tradeoff is
acceptable depends on the use case; copeca surfaces it, it does not judge it.

---

## Token breakdown

Copeca computes cost from four token categories, each with its own price
in the runner's pricing table:

| Token type | What it counts |
|---|---|
| `input_tokens` | Prompt tokens sent to the model |
| `output_tokens` | Tokens generated by the model |
| `cache_creation_tokens` | Tokens written to the prompt cache |
| `cache_read_tokens` | Tokens read from the prompt cache |

Total cost per run:
```
total_cost_usd = input_tokens × price_input
               + output_tokens × price_output
               + cache_creation_tokens × price_cache_creation
               + cache_read_tokens × price_cache_read
```

If a runner does not report a token category, its count is zero and does not
contribute to cost. Staleness warnings fire if the pricing table is older than
30 days.

## Adversarial flags

Flags that highlight when a run's cost or result is suspicious. They are
computed from the parsed output — no flag means nothing suspicious was
detected.

| Flag | What it catches | How it's computed |
|---|---|---|
| `token_snowball` | The agent burns tokens on later turns far more than early ones | max per-turn tokens > num_turns × avg(first 3 turns) × factor |
| `talkative_failure` | The agent produced a lot of output but was still wrong | output_tokens > threshold AND correct == false |
| `tool_storm` | The agent called tools excessively without converging | num_tool_calls > threshold |
| `budget_exhausted` | The agent hit the cost budget before producing a result | cost >= budget AND result is missing or empty |
| `timeout` | The agent exceeded the time limit | duration_ms >= timeout_seconds × 1000 |

Thresholds are currently hardcoded; making them configurable per scenario is
planned. Flags are null (not false) when the runner doesn't provide the data
needed to compute them.
